
# sub-proxy 功能优化需求

## 背景
我现在购买了一个vpn节点服务，他有一个订阅链接，导入到clash后里面有很多国家的节点，默认按延迟自动切换，每次代理可能会用不同国家的节点，价格很便宜，但是问题是这些节点是多人共享的，有些平台会检测你的IP，比如海外社媒会出现封号问题，chatGPT限制只在某些国家能打开，所以针对这些场景需要固定的静态住宅IP。为了处理这个问题，我现在并没有用这个服务，而是单独购买了一个香港的VPS，在上面安装了3x-ui创建了一个vless节点（用于日常国外访问，速度最快），另外购买了一个洛杉矶的住宅IP（用于需要固定IP的场景），在3x-ui中配置为链式代理。最后配置了一个clash的yaml配置文件，里面定义了一些分流规则（哪些走香港，哪些走洛杉矶，哪些直连），在软路由的openclash里导入后使用的，同时软路由里还针对内网ddns做了特殊的过滤，这样我在家的时候就不需要再上网的设备上再运行clash这类代理软件了，只有在外面才需要单独在设备上打开代理软件、且不需要为某些网站手动切换节点。
现在这个yaml配置文件我是用一个自制的后台管理系统在统一维护的，这样所有设备的代理软件里只需要填写同一个订阅地址即可使用，后期维护方便，目前用着也没大问题。但是就是香港vps太贵了，且有时候延迟比较大，不像之前订阅的那种多节点服务那样可以自动切换到最快的节点，导致有时代理非常慢，进而导致后面的链式代理更慢。
我现在想，能不能在不够买香港VPS，只够买这种多国家节点+洛杉矶静态住宅IP的情况下，实现我现在的软路由代理和统一管理代理配置的效果，因为这样更便宜，且默认的国外流量自动选择最快的国家节点，如果是需要美国固定IP的，自动通过当前最快的这个国家节点链式代理过去，这样就可以保证链路最快，同时需要能随时更新或替换多节点服务（因为他们可能不稳定或者过期），在clash 的yaml里该如何配置，是否能兼容openclash、clashverge、shadowrockets这类代理软件。

## 目前的 yaml 配置

当前采用 VPS 作为第一跳，节点单一，延迟较高。配置文件见 proxy-config/vps-proxy.yaml

## 功能优化

> 多节点自动测速选优 + 特定流量强制走固定静态住宅 IP（链式代理）

这是一个非常经典且高阶的代理架构优化需求。你的核心痛点是：**如何在保留“多节点自动测速选优”的同时，实现“特定流量强制走固定静态住宅 IP（链式代理）”，并且要兼容软路由（OpenClash）和客户端（Clash Verge/Shadowrocket），还要方便统一管理。**

答案是：**完全可以实现**。你不需要购买昂贵的香港 VPS 作为中转，可以直接利用你购买的“多国家节点订阅”作为第一跳（中继），洛杉矶静态住宅 IP 作为第二跳（出口）。

以下是具体的实施方案、YAML 配置逻辑以及兼容性说明。

### 核心原理：中继代理 (Relay Proxy)

Clash 内核（包括 Meta/Mihomo 内核，这是目前 OpenClash 和 Clash Verge 的主流内核）支持 `relay` 类型的代理组。
*   **传统链式**：手动指定 `节点 A` -> `静态 IP`。如果 `节点 A` 慢了，整体就慢了。
*   **你的新方案**：定义一个策略组 `优选加速节点`（自动测速选最快），然后定义一个 `relay` 组，其路径为 `优选加速节点` -> `洛杉矶静态 IP`。
*   **效果**：所有需要美国固定 IP 的流量，会自动先连接到当前延迟最低的那个多国节点，再跳转到洛杉矶 IP。既保证了出口 IP 固定，又保证了入口链路最优。

### 第一步：准备工作与架构设计

你需要维护两个来源的配置：
1.  **来源 A（动态）**：你购买的多国家节点订阅链接（机场）。
2.  **来源 B（静态）**：你自己编写的包含洛杉矶静态 IP 和分流规则的配置文件。

**在自制的后台管理系统中，你需要做以下逻辑处理：**
1.  定期拉取 **来源 A** 的节点列表。
2.  将 **来源 B** 中的静态节点配置硬编码进去。
3.  生成最终的 `config.yaml` 供内网设备订阅。

---

### 第二步：YAML 配置文件详解

以下是一个标准的、兼容 OpenClash 和 Clash Verge (Mihomo 内核) 的配置模板。请将其集成到你的后台生成逻辑中。

```yaml
# ================= 基础设置 =================
port: 7890
socks-port: 7891
redir-port: 7892
tproxy-port: 7893
allow-lan: true
mode: rule
log-level: info
external-controller: 0.0.0.0:9090
secret: "" # 建议设置密码
dns:
  enable: true
  listen: 0.0.0.0:53
  enhanced-mode: fake-ip
  fake-ip-range: 198.18.0.1/16
  nameserver:
    - 223.5.5.5
    - 114.114.114.114
  fallback:
    - https://dns.cloudflare.com/dns-query
    - https://dns.google/dns-query
  fallback-filter:
    geoip: true
    ipcidr:
      - 240.0.0.0/4

# ================= 代理节点定义 =================
proxies:
  # 1. 这里填入你的【洛杉矶静态住宅 IP】配置 (手动维护或后台写入)
  # 假设你使用的是 VLESS 协议 (根据你 3x-ui 的描述，如果是买的纯 IP 可能是 SOCKS5，请按实际修改)
  # 如果是买的纯静态 IP (SOCKS5/HTTP)，配置如下：
  - name: "LA-Static-Residential"
    type: socks5 
    server: 你的洛杉矶静态 IP 地址
    port: 端口号
    username: 用户名
    password: 密码
    udp: true
    # 如果是 VLESS/Trojan 等协议，请填写对应的完整配置字段
  
  # 2. 这里通过脚本动态插入【多国家节点订阅】的内容
  # 【后台系统逻辑】：在此处拼接从机场订阅解析出来的 proxies 列表
  # 例如：
  # - name: "US-Auto-Node-1" ...
  # - name: "JP-Auto-Node-2" ...
  # - name: "SG-Auto-Node-3" ...

# ================= 代理提供商 (可选，如果你不想解析订阅，直接用 Proxy Provider) =================
# 如果你的后台系统不方便解析 YAML，可以直接使用 proxy-providers 功能引用原始订阅链接
# 但为了统一管理和添加静态节点，建议在后台解析后合并到上面的 proxies 列表中更稳妥。
# 此处演示直接合并到 proxies 的方案，兼容性最好。

proxy-groups:
  # 1. 【核心】自动选择最快的多国节点 (作为中继的第一跳)
  - name: "🚀 自动优选加速"
    type: url-test
    use: [] # 如果用了 proxy-providers 则填名称，否则留空，直接使用下方 proxies 过滤
    proxies: [] 
    # 【后台逻辑】：在这里动态填入所有从机场订阅获取的节点名称
    # 或者使用 filter 正则匹配（推荐）：
    # 注意：url-test 组通常不支持直接正则过滤子节点，需要在生成配置时把节点名填进去。
    # 更好的方式：在下面定义一个包含所有机场节点的组，然后被 relay 引用。
    url: http://www.gstatic.com/generate_204
    interval: 300
    tolerance: 50

  # 2. 【关键】链式代理组：自动优选 -> 洛杉矶静态 IP
  # 这种类型叫 relay，只有 Mihomo/Meta 内核支持 (OpenClash 默认开启 Meta 内核即可)
  - name: "🇺🇸 美国固定 IP (链式)"
    type: relay
    proxies:
      - "🚀 自动优选加速"  # 第一跳：自动选最快的机场节点
      - "LA-Static-Residential" # 第二跳：固定的洛杉矶出口

  # 3. 其他常规组
  - name: "🌍 国外流量自动选"
    type: select
    proxies:
      - "🚀 自动优选加速"
      - "🇺🇸 美国固定 IP (链式)"
      - DIRECT

  - name: "🎯 国内直连"
    type: select
    proxies:
      - DIRECT
      - "🚀 自动优选加速"

# ================= 规则定义 (Rules) =================
rules:
  # 1. 需要固定美国 IP 的场景 (ChatGPT, 特定社媒账号等)
  # 这里的域名需要根据你的实际需求补充
  - DOMAIN-SUFFIX,openai.com,🇺🇸 美国固定 IP (链式)
  - DOMAIN-KEYWORD,chatgpt,🇺🇸 美国固定 IP (链式)
  - DOMAIN-SUFFIX,tiktok.com,🇺🇸 美国固定 IP (链式)
  - DOMAIN-SUFFIX,instagram.com,🇺🇸 美国固定 IP (链式)
  - DOMAIN-SUFFIX,facebook.com,🇺🇸 美国固定 IP (链式)
  - DOMAIN-SUFFIX,twitter.com,🇺🇸 美国固定 IP (链式)
  
  # 2. 国内流量直连
  - GEOIP,CN,🎯 国内直连
  
  # 3. 其余国外流量走自动优选 (速度最快)
  - MATCH,🌍 国外流量自动选

# ================= 脚本生成提示 =================
# 在你的后台系统中：
# 1. 读取机场订阅 -> 解析出 nodes 列表。
# 2. 将 nodes 列表写入 yaml 的 `proxies` 部分。
# 3. 提取所有 nodes 的 `name`，填入 `🚀 自动优选加速` 组的 `proxies` 列表中。
# 4. 保持 `LA-Static-Residential` 不变。
# 5. 输出最终 yaml。
```

### 第三步：关键技术点与兼容性说明

#### 1. 内核兼容性 (非常重要)
*   **Relay 类型支持**：`type: relay` 是 **Clash Meta (Mihomo)** 内核特有的功能。
    *   **OpenClash**: 必须在插件设置中将“核心版本”切换为 **Meta (Mihomo)** 核心。默认的 Premium 或 Dev 核心可能不支持 `relay` 或者支持不完善。
    *   **Clash Verge Rev / Clash Nyanpasu**: 默认使用的就是 Mihomo 内核，完美支持。
    *   **Shadowrocket (iOS)**:
        *   Shadowrocket 本身不直接支持 `relay` 类型的 *组* 定义（它主要支持策略组选节点）。
        *   **解决方案**：在 iOS 上，你通常不能依赖云端的 `relay` 组逻辑。
        *   **变通方法**：对于 Shadowrocket，你可以在规则里直接写死。但为了实现“自动选最快 + 固定出口”，在 iOS 端比较难完美实现全自动链式。
        *   **推荐做法**：在 Shadowrocket 中，你可以手动创建一个“策略组”，类型为 `URL-Test` (包含所有机场节点)，然后再创建一个手动选择的节点指向洛杉矶 IP？不行，SR 不支持自动链式。
        *   **针对 iOS 的最佳实践**：既然你有软路由，**主力设备（手机/电脑）在家时连接 Wi-Fi，流量由软路由（OpenClash Meta）处理**，此时 `relay` 生效，体验完美。当你出门使用蜂窝网络或陌生 Wi-Fi 时，如果在手机上用 Shadowrocket，你可能需要接受稍微手动的操作，或者在订阅配置中为 iOS 单独生成一份配置（不使用 relay，而是让用户手动选择“美国固定”节点，该节点在配置里被硬编码为某个特定的优质机场节点+静态 IP，但这失去了自动测速意义）。
        *   **结论**：这套方案在 **软路由 (OpenClash Meta)** 和 **电脑端 (Clash Verge)** 上是完美的。对于 **手机端 (Shadowrocket)**，建议在家连 Wi-Fi 享受软路由成果；在外若必须用固定 IP，可能需要在手机软件里手动切换到指定的“美国固定”节点（如果你在配置里把“美国固定”做成一个普通的节点而不是组，那就没法自动测速了）。
        *   **进阶技巧**：其实可以在配置里把 `🇺🇸 美国固定 IP (链式)` 这个组，在发给 Shadowrocket 时，利用 Clash 的特性，有些第三方客户端能识别。如果不能，就在家靠软路由，出门靠手动切换，或者出门只追求速度不追求固定 IP。

#### 2. 关于“多节点服务不稳定/过期”的处理
由于你使用了自制后台：
*   当机场订阅链接失效或节点变动时，你的后台脚本重新拉取订阅。
*   只要新的订阅里有节点，后台脚本就会更新 `proxies` 列表和 `🚀 自动优选加速` 组的成员列表。
*   **无需人工干预**：只要后台脚本逻辑健壮（比如捕获异常，如果拉取失败则保留旧配置），系统就能自动适应机场的变化。

#### 3. 性能优化
*   **延迟问题**：之前你觉得慢，是因为固定了一个可能很慢的香港 VPS 做第一跳。现在第一跳变成了“自动优选”，系统每 5 分钟（`interval: 300`）测试一次所有机场节点，永远选最快的一个作为入口。
*   **链路**：`本地` -> `最快的新加坡/日本/美国机场节点` -> `洛杉矶静态 IP` -> `目标网站`。这条链路通常比 `本地` -> `固定香港 VPS` -> `洛杉矶` 要快得多，因为机场节点通常带宽大、线路优化好。

### 第四步：实施流程总结

1.  **升级软路由内核**：登录 OpenClash 后台，确保安装并启用了 **Meta (Mihomo)** 内核。
2.  **改造后台脚本**：
    *   编写 Python/Go/Node.js 脚本。
    *   输入：机场订阅 URL + 洛杉矶静态 IP 参数。
    *   处理：下载订阅 -> 解析 YAML -> 提取节点 -> 组装新的 YAML（加入静态节点、创建 Relay 组、填充规则）。
    *   输出：将生成的 `config.yaml` 托管在你的 Web 服务器上，提供一个稳定的 HTTP 订阅链接。
3.  **配置客户端**：
    *   **软路由 (OpenClash)**：订阅链接指向你的后台生成的 URL。模式设为 `Rule`。
    *   **电脑 (Clash Verge)**：同上。
    *   **手机 (Shadowrocket)**：同上。*注：如果 SR 不识别 relay 组导致无法上网，需在后台判断 User-Agent，如果是 SR，则降级处理（例如不提供 relay 组，只提供普通节点，让用户出门手动选，或者在家连 Wi-Fi 忽略此问题）。但在大多数新版基于 Mihomo 的客户端中，Relay 是支持的。Shadowrocket 对 Clash 配置的支持有限，可能无法完美运行 Relay 组，建议手机在外网时不强求“自动测速+固定 IP”同时满足，或者仅在家使用此功能。*
4.  **规则微调**：根据你的实际需求，在 `rules` 部分精确添加需要走 `🇺🇸 美国固定 IP (链式)` 的域名（如 `openai.com`, `claude.ai`, 特定的电商网站等）。

### 常见问题排查 (FAQ)

*   **Q: 为什么配置后无法上网？**
    *   A: 检查 OpenClash 是否启用了 Meta 内核。检查 `relay` 组里的两个节点名字是否与 `proxies` 里定义的完全一致（大小写敏感）。
*   **Q: 静态 IP 被封了怎么办？**
    *   A: 只需要在你的后台管理系统中修改洛杉矶静态 IP 的配置参数，重新生成配置文件，所有设备下次更新订阅（或手动点击更新）即可生效，无需逐台修改。
*   **Q: 机场节点太多，配置文件太大？**
    *   A: 可以在后台脚本中加入过滤逻辑，例如只保留延迟低于 500ms 的节点，或者只保留特定国家的节点放入 `🚀 自动优选加速` 组，减少负载。

通过这种 **“动态优选入口 + 静态固定出口”** 的架构，你既省去了昂贵的香港 VPS 费用，又解决了共享 IP 封号的问题，同时利用了廉价机场的高速带宽，是目前性价比最高的解决方案。
